## 팀의 AI 개발 도구 세팅을 주도

`.mcp.json`을 프로젝트에 추가한 사람이 김도혁입니다 ([커밋: "chore: Context7, playwright, next 개발 도구 mcp 추가 적용", 2026-06-21]). 이 설정으로 Claude Code에 세 가지 MCP 서버가 연결됩니다.

- **context7** — React·Next.js·Supabase 등 라이브러리의 최신 문서를 실시간으로 조회. 버전이 자주 바뀌는 Next.js 16 App Router 같은 스택에서 학습 데이터가 오래된 답변 대신 실제 문서를 참조하게 해줍니다.
- **playwright** — 브라우저를 직접 조작해 E2E 시나리오를 검증. `review/e2e/` 테스트 인프라와 맞물려, AI가 직접 화면을 띄워보고 동작을 확인하는 용도로 쓸 수 있습니다.
- **next-devtools** — Next.js 프로젝트 전용 디버깅 도구.

## Claude Opus 플랜 모드로 재설계 요청

모바일 헤더 스크롤 숨김 기능을 고칠 때(`docs/작업기록/20260617-header-hide-on-scroll.md`), 기존 `IntersectionObserver` 방식의 구조적 한계를 발견하고 **"Opus 플랜 모드로 재설계를 요청"**했다고 작업기록에 직접 남겼습니다. 패치보다 재설계가 필요하다고 판단했을 때, 더 깊이 사고하는 모델(Opus)과 실행 전에 계획부터 세우는 플랜 모드를 의도적으로 선택한 사례입니다.

## AI 기능 자체를 설계한 경험이 AI 활용에도 녹아있다

[AI 모듈 4종 통합 설계](/features/김도혁)를 주도하며 Gemini → Claude로 제공자를 전환하고, 서버 사이드 전용 호출·rate-limit·3계층 구조 같은 원칙을 세운 경험이 있습니다. AI를 "제품 기능"으로 설계해본 사람은 AI를 "개발 도구"로 쓸 때도 프롬프트에 무엇을 맡기고 무엇을 직접 검증할지 경계를 더 분명히 긋는 경향이 있는데, 실제로 어떤 부분을 Claude Code에 맡기고 어떤 부분을 직접 검토했는지는 본인이 채워주시면 좋겠습니다.
