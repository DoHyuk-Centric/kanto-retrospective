## 팀에 Claude Code 행동 지침을 도입

`CLAUDE.md`(behavioral guidelines)를 프로젝트에 처음 추가한 사람이 임태형입니다 ([커밋: "chore: add CLAUDE.md for Claude Code behavioral guidelines", 2026-06-17]). 이 문서는 지금 이 회고 사이트의 루트에도 그대로 이어져 있는, "가정하지 말고 확인하라", "필요한 만큼만 고쳐라" 같은 원칙을 담고 있습니다. 팀이 AI를 도구로 쓰기 시작한 시점에 어떤 규율을 세우고 싶었는지가 드러나는 지점입니다.

## 대화로 네 번 바뀐 설계 — AI와의 반복 논의 흔적

AI 이미지 검열 기능을 구현할 때(`docs/작업기록/20260625-이미지검수AI-구현.md`), 작업기록에 이렇게 남겼습니다.

> "설계 결정이 사용자와의 대화를 통해 여러 차례 바뀌었다."

독립 마이크로서비스 → API 라우트 내장, fail-open → fail-closed, OpenAI 폴백 검토 후 폐기 → Groq로 교체, 이중화 → 단일 제공자로 단순화까지 네 단계를 거쳤습니다. 이건 한 번에 완성된 설계가 아니라, AI와 주고받으며 매번 "이게 맞나?"를 검증하고 방향을 바꾼 흔적입니다. 특히 "fail-open이 싫다"는 피드백을 반영해 안전 우선으로 설계를 뒤집은 부분은, AI가 제안한 효율적인 기본값(실패 시 통과)을 그대로 받아들이지 않고 서비스 특성에 맞게 직접 판단해 되돌린 사례로 읽힙니다.

## 구조화된 로그를 읽어 정확히 진단

칸토Go 작업 중 발견한 런타임 버그 3건을 두고 이렇게 기록했습니다.

> "`.next/dev/logs/next-development.log`(JSON Lines)를 읽어서 정확한 에러 코드와 타임스탬프를 확인하는 방식으로 3건 모두 빠르게 특정."

추측 대신 구조화된 로그 파일을 직접 읽어 근거를 확보하는 방식은, AI에게 "이거 왜 안 돼?"라고 막연히 묻는 대신 로그라는 확인 가능한 자료를 던져주고 진단을 받는 접근과 맞닿아 있습니다.
