전체 커밋의 약 40%(433건)를 차지하며 관리자 대시보드(144건), 구인구직(126건), 공통 컴포넌트(147건), 방렌트(93건), 중고거래(69건), 프로필(67건), 메인(50건)까지 폭넓게 걸쳐 있습니다. 특히 AI 기능 통합과 자체 알고리즘(KTS/KPPS) 설계·구현을 주도했습니다.

## Kanto Trust Score(KTS) — "망고지수"

[원본: docs/설계/kanto-trust-score(KTS).md, kts-kpps-implementation.md]

기존엔 `avg_rating`(후기 별점 평균) 하나로만 사용자 평판을 표현했습니다. 후기가 없는 신규 유저와 사기 전력이 있는 유저를 구분할 방법이 없었고, 필리핀 현지 거래 특성상 신뢰를 빠르게 판단할 지표가 필요했습니다. 당근마켓의 "당근온도"에서 영감을 받았지만 "온도"라는 표현은 차용하지 않고 "점수"로 차별화했습니다.

`KTS = clamp(36 + ReviewScore(±25) + TransactionScore(0~15) + ActivityScore(0~10) + TrendScore(0~14) − PenaltyScore, 0, 100)` 라는 다요소 점수로 설계했습니다. 시작 점수를 36점으로 잡아 신규 유저가 최하위로 시작하지 않게 했고, 제재와는 완전히 독립적으로 운영해서 KTS가 높아도 위반하면 동일하게 제재받습니다. 후기 1건만으로 만점을 받는 걸 막기 위해 후기 수에 따른 confidence_weight(20건 이상이어야 1.0)를 도입했고, TrendScore는 `user_trust_history` 주간 스냅샷 테이블로 최근 4주간의 등급 변화 패턴(상승세/안정/간헐적/하락세)까지 반영합니다. 계산은 실시간이 아니라 하루 1~2회 배치(pg_cron)로 처리해 비용과 일관성 사이에서 트레이드오프를 택했습니다.

## Kanto Popular Post Score(KPPS) — 어뷰징에 강한 인기 게시물 선정

[원본: docs/설계/kanto-popular-post-score(KPPS).md]

기존 인기 게시물 기준은 `like_count >= 20` 단일 임계값이라 조회수 어뷰징에 취약했고, 시간 요소나 작성자 신뢰도, 게시물 품질을 전혀 반영하지 못했습니다. KTS와 연계해서, 신뢰도 낮은(E등급) 작성자는 KPPS 점수와 무관하게 인기 선정에서 강제로 배제하는 게이트를 뒀습니다.

조회수만 높고 좋아요가 없는 어뷰징을 막기 위해 EngagementScore(참여 절대량)와 LikeRateBonus(좋아요/조회수 전환율)를 분리 설계했는데, 조회 2000·좋아요 1건 같은 사례에서 LikeRateBonus가 사실상 0으로 수렴하는 걸 직접 검증했습니다. 방렌트는 장기 노출 특성을 고려해 RecencyBonus 최솟값을 3점으로 보호하고, 중고거래는 1점으로 둬서 빠른 순환을 유도했습니다. `recalculate_kpps()`는 반드시 `recalculate_kts()` 배치(03:00) 완료 후(03:30)에만 실행되도록 pg_cron으로 순서를 강제했습니다. 구인구직은 KPPS에 필요한 댓글 데이터가 없어 기존 `popular_count`를 그대로 두고 KPPS를 적용하지 않았습니다.

실제 구현 단계에서는 설계와 다른 지점도 정직하게 기록했습니다 — 제재 타입 `perm`(영구)은 설계 문서에 감점 기준이 정의되지 않아 "미반영"으로 남겼고, 테스트용으로 남겨둔 "지금 갱신" 버튼과 `refreshKtsScore` 서버 액션은 배포 전 제거 체크리스트에 올려뒀습니다.

## AI 기능 4종 통합 — 이미지 검열, 챗봇, 요약, 관리 보조

[원본: docs/설계/ai-module-design.md, AI챗봇_설계문서.md]

플랫폼에 콘텐츠 검열이나 스팸/사기 탐지, 요약, 관리자 보조 기능이 전무했던 상태에서 Anthropic Claude API(`claude-haiku-4-5-20251001`) 기반으로 4개 기능을 한 아키텍처 원칙으로 통일해 설계했습니다. 서버 사이드 전용 호출(클라이언트는 API 라우트로만 간접 접근), 기존 Upstash Redis rate-limit 패턴 재사용, 서비스→API 라우트→훅/컴포넌트 3계층 구조를 고수했습니다.

이미지 검열은 파싱이 실패하면 `{safe:true}`로 처리해, 애매하면 통과시키는(false positive 방지 우선) 방향을 택했습니다. 챗봇은 `anthropic.messages.stream()`으로 `ReadableStream` 청크 전송, 비용 절감을 위해 최근 10개 메시지만 전송합니다. 게시글 관리 AI는 한 번의 호출로 `safe`/`isSpam`/`isFraud`를 동시에 판단해 `approve`/`warn`/`reject`를 산출하지만, 관리자 전용 심층분석 API는 신뢰도 점수(0~1)까지만 반환하고 최종 결정은 AI가 아니라 사람이 하도록 설계했습니다. 흥미로운 지점은 같은 프로젝트 안에서 초기 챗봇 설계(Gemini)에서 이 통합 설계(Claude)로 AI 제공자가 전환된 이력이 문서에 그대로 남아있다는 것입니다.

챗봇 백엔드는 `POST /api/ai/chat`이 SSE로 스트리밍 응답하고, Gemini 2.0 Flash의 100만 토큰 컨텍스트를 활용해 별도 RAG 파이프라인 없이 지식베이스 문서 전체를 시스템 프롬프트로 통째 주입하는 단순화된 구조를 택했습니다. 대화 이력은 서버/DB가 아니라 브라우저 sessionStorage에 저장해, 탭을 닫으면 초기화되지만 비용이 없고 로그인이 불필요하다는 트레이드오프를 명시적으로 감수했습니다.

## 1:1 실시간 채팅 시스템

[원본: docs/설계/chat-system.md]

게시글(중고거래/방렌트/구인구직)을 매개로 한 1:1 채팅에서 커서 기반 페이지네이션을 택했습니다 — 오프셋 방식은 새 메시지가 쌓일수록 페이지가 밀려 중복·누락이 생기기 때문입니다. unread 카운트는 messages 전체를 JOIN해서 세는 대신 DB 컬럼(`user_id_1_unread`/`user_id_2_unread`)으로 관리해 쿼리 비용을 거의 없앴는데, 동시성 문제(두 사람이 동시에 보내면 카운트가 누락될 수 있음)를 인지하고도 1:1 채팅 특성상 발생 가능성이 낮다고 판단해 "허용 범위"로 남겼습니다. 도배 방지 윈도우는 처음 1.5초로 설정했다가 정상적으로 빠르게 타이핑하는 사용자까지 막혀서 3초로 튜닝했습니다.

Supabase Realtime `postgres_changes` 구독으로 별도 서버 인프라 없이 실시간성을 확보했고, 낙관적 UI(임시 ID `Date.now()`) 후 Realtime INSERT로 교체하는 패턴을 썼습니다. 스크롤 복원은 `requestAnimationFrame`으로 다음 페인트 직전에 재계산해 `setState` 직후 DOM 미반영 문제를 피했습니다.

## 검색/필터 아키텍처, SEO, 약관 캐싱

구인구직·중고거래·방렌트 세 도메인의 검색·필터·페이지네이션을 서버 컴포넌트가 `searchParams`를 읽어 DB 필터링하고 클라이언트는 URL만 갱신하는 통일된 패턴으로 정리했습니다([원본: docs/설계/url-search-filter-architecture.md]). SEO는 robots.ts/sitemap.ts/OG 이미지/JSON-LD까지 전반을 구현했고([원본: docs/설계/seo-metadata-optimization.md]), 이 부분은 14차 리뷰에서 "다섯 조 중 SEO 인프라가 가장 단단하다"는 평가를 받았습니다.

약관 페이지(`/terms/[type]`)는 Notion API 연동에서 탭을 이동할 때마다 API가 재호출돼 로딩이 반복되는 문제를, `generateStaticParams`(next-intl의 `cookies()` 때문에 불가) → `staleTimes`(실시간 페이지까지 영향 → 기각) → URL 기반 i18n 전환(범위 초과 → 기각) 세 가지를 차례로 검토하고 기각한 끝에, 서버 렌더링을 포기하고 localStorage + TTL(10시간) 클라이언트 캐싱으로 전환했습니다([원본: docs/설계/terms-notion-caching.md]). SEO 저하라는 트레이드오프를 "약관 페이지 특성상 영향 없음"으로 판단해 감수한 결정입니다. (이후 성능검사에서 이 구조의 한계가 다시 드러나 [성능개선](/performance)에서 SSR+ISR로 재전환됩니다.)

## 프로필, 회원탈퇴

프로필 페이지는 클라이언트 마운트 후 fetch하면 탭 전환마다 로딩 지연이 생기는 문제를 피하려고 서버 컴포넌트가 먼저 조회한 값으로 훅의 `useState`를 초기화하는 패턴을 택했습니다([원본: docs/설계/profile-page-design.md]). 회원탈퇴는 즉시 완전 삭제 대신 30일 유예를 둔 소프트 삭제로 설계했고, `users.deleted_at` 컬럼 하나로 상태를 표현하면서 FK 참조 순서를 지키는 다단계 캐스케이드 삭제 로직(`cleanup_deleted_users()`)을 만들었습니다([원본: docs/설계/account-deletion-design.md]). 찜하기 기능은 클라이언트 차단만 적용돼 있어 "DevTools로 우회 가능하다"는 한계를 스스로 문서에 명시했습니다.
