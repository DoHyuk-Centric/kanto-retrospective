243건의 커밋 중 구인구직(44건), 중고거래(40건), 관리자(38건), 방렌트(33건), 공통 컴포넌트(95건), 칸토Go 관련 컴포넌트(22건)에 걸쳐 있습니다. 가장 뚜렷한 단독 기능은 다국어(i18n) 지원입니다.

## 3개 언어를 쿠키 하나로 — next-intl 다국어 지원

[원본: docs/설계/internationlization.md]

필리핀 현지 서비스라는 특성상 한국어·영어·필리핀어 다국어 지원이 필수였습니다. next-intl의 표준적인 `[locale]` URL 세그먼트 방식 대신 **쿠키 기반**을 선택했는데, 이미 존재하던 `(admin)`/`(user)` 라우트 구조와 Supabase 미들웨어를 건드리지 않기 위해서였습니다. DB에 저장되는 사용자 콘텐츠(게시글 제목·내용, 채팅 메시지, 작성자 이름)는 번역 대상에서 제외하고, "정적 UI 틀"만 번역 대상으로 명확히 한정했습니다. 필리핀어(`fil`)는 상대시간 표현이 지나치게 길어져 UI가 깨지는 문제가 있어, 상대시간만 영어(`en-US`)로 별도 포맷하는 예외를 뒀습니다.

가장 트레이드오프가 뚜렷했던 결정은 언어 전환 방식입니다. 서버 액션으로 구현했을 때 Supabase 미들웨어의 `Set-Cookie` 타이밍과 얽혀 `router.refresh()`에 새 쿠키가 확정적으로 반영되지 않는 문제가 있어서, 결국 **클라이언트에서 `document.cookie`를 직접 설정**하는 방식으로 우회했습니다. 그런데 이번엔 React Compiler의 immutability 규칙(`react-hooks/immutability`) 때문에 컴포넌트 본문에서 `document.cookie`를 직접 바꾸는 게 ESLint 에러로 걸려서, 부수효과를 컴포넌트 밖 함수로 분리해야 했습니다. `RENT_TYPES`처럼 DB 저장값과 결합된 상수는 원문(한국어)을 canonical 키로 유지하고 표시 시점에만 매핑 번역하는 방식으로, DB 마이그레이션 없이 다국어를 지원했습니다.

번역 누락을 잡는 것도 간단하지 않았습니다. `MISSING_MESSAGE` 콘솔 경고만으로는 하드코딩된 텍스트가 남아있는 걸 잡아낼 수 없어서, 별도의 전수 감사(batch)가 필요했습니다. [문제해결 페이지](/troubleshooting/이동근)에 기록했듯, 로케일 코드로 `ta`(타밀어)를 잘못 쓰고 있던 걸 `fil`로 바로잡은 것도 이 작업의 연장선이었습니다.

## 목록 페이지들과 칸토Go 컴포넌트

구인구직·중고거래·방렌트 목록 페이지 작업과 칸토Go의 컴포넌트 레이어(22건)를 주로 맡았습니다. 사이트 전체를 가로지르는 성능 감사(채팅 제외)를 진행해 상세 페이지의 직렬 DB 왕복 문제, 목록 페이지의 비효율적인 페이지네이션 문제를 짚어낸 것도 이동근입니다 — 이 부분은 [문제해결](/troubleshooting/이동근)과 [성능개선](/performance) 양쪽에 걸쳐 있습니다.
