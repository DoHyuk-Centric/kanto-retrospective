128건의 커밋으로 팀 내 가장 적은 커밋 수지만, 설계 문서로는 가장 많은(8건) 기능을 남겼습니다. 칸토Go(위치 기반 번개모임)라는 신규 서비스 축 전체를 기획부터 구현까지 주도했고, 결제·신뢰·관리자 영역의 굵직한 설계를 맡았습니다.

## 칸토 GO! — 지도 위의 번개모임을 기획하다

[원본: docs/설계/20260622-칸토go-번개모임-기획.md, 20260623-칸토go-최종정리.md]

기존 Kanto는 중고거래·구인구직·렌탈 중심의 정적 게시물 서비스였습니다. 필리핀 현지에서 위치 기반으로 가볍게 모이고 싶은 니즈를 충족하기 위해, 지도 위에서 핀을 눌러 참여하는 완전히 새로운 서비스 축을 기획했습니다.

지도 SDK는 reverse geocoding과 필리핀 커버리지를 고려해 Google Maps로, 최종적으로는 `@vis.gl/react-google-maps`로 확정했습니다. 모임 상세 UI를 두고 기획자안(bottom sheet)과 디자이너안(우측 슬라이드 패널)이 충돌해 우측 슬라이드 패널로 결정했고, Realtime 적용 시점도 기획자안(v2부터)과 디자이너안(v1부터) 사이에서 v1부터 포함하는 쪽으로 재확정했습니다. 채팅은 단계적으로 분리해서 v1에서는 그룹채팅을 빼고, v2에서 도입한 뒤 모임 종료 후 24시간 유예를 두고 자동으로 완전삭제(복구 불가)되도록 설계했습니다. 모임 종료 처리는 별도 크론 없이 쿼리 시점에 `end_at > now()` 필터로만 판단해서 구현을 단순화했습니다.

데이터 모델은 `posts`(post_type='meetup') + `meetups`(위치/시간/정원) + `meetup_participants`(참여자, unique 제약)로 구성했고, 정원의 88% 이상이면 "마감임박" 뱃지를 붙였습니다. 실제 구현 과정([문제해결 페이지](/troubleshooting/임태형)에 상세)에서는 디자이너가 AI로 생성한 코드 패키지의 여러 가정 오류(UUID 가정 vs 실제 bigint, 잘못된 import 경로)를 코드베이스 기준으로 하나씩 바로잡아야 했습니다.

## Xendit 안전결제(에스크로)와 그 이후

[원본: docs/설계/xendit-payment-escrow.md, transaction-complete-system-message.md, transaction-review-write.md]

중고거래가 채팅으로 협의 후 진행되는 구조라, 사기 방지를 위해 제3자(Xendit)가 결제금을 임시 보관했다가 수령 확인 후 정산하는 흐름을 설계했습니다. 테스트 모드 기준이라 실제 자금 보관·정산(disbursement)은 구현하지 않고 상태값으로만 시뮬레이션한다는 한계를 스스로 명시했고(이후 [박소유가 실제 Disbursement 연동을 완성](/features/박소유)), 결제완료 처리는 webhook 단일 경로에 의존하지 않고 webhook + return 페이지 폴백으로 이중화했습니다.

결제 상태가 바뀌어도 채팅창의 대화가 이어지면 결제 카드가 스크롤로 밀려 올라가 사용자가 "결제완료" 같은 상태 전환을 놓치는 문제를 실사용 중 발견하고, 시스템 메시지를 자동으로 채팅창에 삽입하는 기능을 추가했습니다. 이 과정에서 `permission denied for table messages`라는, `transactions`에는 있지만 `messages`에는 없던 service_role 권한 문제를 만나 `GRANT`로 해결했습니다.

거래 후기 기능은 원래 스켈레톤만 있던 걸 실제로 구현하면서 다섯 가지 버그를 연달아 만났습니다 — 중복 판정 단위를 "상품당 1회"로 뒀다가 같은 상품에 여러 거래가 있으면 후기 배너가 영구 숨김되는 걸 발견해 "거래당 1회"로 바꿨고, 후기 진입점을 결제 카드 안에 두면 대화가 밀리는 문제를 겪은 뒤 입력창 위 고정 배너로 옮겼고, 메시지 페이지네이션 때문에 결제 카드가 최신 50개 밖으로 밀려나면 배너 대상 거래를 못 찾는 문제를 서버 권위 조회로 해결했고, `released_at DESC` 단독 정렬이 nullable 컬럼 동순위에서 비결정적이라 후기 작성 후에도 배너가 재노출되는 버그를 `id DESC` 보조 정렬로 잡았습니다.

## 어드민: 신고 처리, 대시보드 명세, 제재 데이터 리팩토링

[원본: docs/설계/admin-reports-actions.md, admin-dashboard-supabase-spec.md, sanction-data-refactoring.md]

신고 검토→처리완료/무시→제재 적용 흐름을 설계하면서, 제재 결과(종류+만료일)가 `users.suspended_until`, `user_sanctions`, `common_reports` 세 곳에 중복 저장돼 정합성이 깨질 위험을 발견했습니다. `user_sanctions`를 단일 진실 원천으로 삼고 **Phase 1(읽기만 전환, 기존 컬럼 유지) → Phase 2(컬럼 DROP)**의 단계별 안전 진행(dual-write) 전략을 채택했는데, 이 과정에서 마이그레이션 이력 divergence(로컬-원격 `schema_migrations` 불일치, 과거 SQL Editor 직접 적용 관행 때문)라는 별도의 인프라 문제까지 진단하고 `supabase migration repair`로 풀어냈습니다.

어드민 대시보드는 지역별 게시글 수처럼 세 테이블에 흩어진 데이터를 Supabase JS 체이닝으로 처리하면 3번 쿼리 후 프론트에서 합쳐야 하는 비효율이 있어, 복잡한 집계를 Postgres RPC 함수 5종(`get_daily_signups` 등)으로 DB 내부에서 처리하도록 명세했습니다.

당시엔 `users.suspended_until`이 DB에 저장만 될 뿐 실제로 글쓰기/댓글을 막는 로직이 없었다는 점, 비활성화된 게시글에 직접 URL로 접근하면 안내 없이 404만 뜨는 문제를 "남은 작업"으로 솔직하게 문서에 남겼습니다.

## AI 이미지 검열 — 네 번 바뀐 설계

[원본: docs/설계/ai_image_moderation.md, docs/작업기록/20260625-이미지검수AI-구현.md]

게시글 이미지 업로드 시 부적절한 콘텐츠(성인/폭력/혐오/불법물)를 막는 기능을 독립 HTTP 마이크로서비스로 설계했습니다 — Kanto 전용이 아니라 타 사이트에서도 REST 호출만으로 재사용 가능하게 하려는 의도였습니다(Hono 프레임워크, Docker 배포). 하지만 실제 구현 단계에서 이 설계는 사용자와의 대화를 거치며 네 번 바뀌었습니다: ① 독립 마이크로서비스는 단일 Next.js 앱에는 과한 인프라 부담이라 API 라우트 내장으로 축소, ② 최초 "실패 시 통과(fail-open)" 설계를 사용자가 "미검수 이미지 공개는 안 된다"며 거부해 fail-closed로 전환, ③ OpenAI 폴백을 검토했으나 무료 플랜에서 지출 한도 설정 자체가 막혀 있어 Groq로 교체, ④ "Gemini 1차 + Groq 2차" 이중화로 운영했으나 Gemini 무료 티어가 하루 20회로 너무 제한적이고 챗봇과 API 키/프로젝트를 공유해 할당량을 나눠 쓰는 문제를 발견해, 결국 Groq 단일 제공자로 최종 단순화했습니다.

이 네 번의 방향 전환을 실측으로 검증하며 "모델별 레이트리밋이 완전히 독립적임을 확인"하는 등, end-to-end 검증 결과를 구체적인 수치로 남겼습니다. 제출 시점 2차 방어나 임계값 튜닝처럼 미구현으로 남긴 부분도 한계로 명시했습니다.
