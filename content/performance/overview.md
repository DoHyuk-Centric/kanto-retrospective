## 방법론

2026-06-30에 성능 감사 프레임워크를 도입하고, 이후 팀원 4명이 담당 페이지를 나눠 측정·개선을 진행했습니다.

**측정 규칙** ([원본: docs/성능/README.md, docs/설계/20260630-성능감사-프레임워크.md])
- **PageSpeed Insights(PSI)** — Lighthouse Lab 데이터 기준. Field(CrUX) 데이터는 참고용으로만 기록
- 반드시 **배포된 URL**만 측정 (로컬 dev 서버 측정 금지), 모바일 + 데스크탑 각각 2회 이상 측정 후 대표값 기록
- 판정 기준: LCP ≤2.5s 🟢 / 2.5~4s 🟡 / >4s 🔴, CLS ≤0.1 🟢, FCP ≤1.8s 🟢, INP ≤200ms 🟢, TTFB ≤800ms 🟢
- 우선순위: **P0**(즉시) = LCP>4s·CLS>0.25·INP>500ms, **P1**(이번 스프린트) = LCP 2.5~4s·API>500ms, **P2**(백로그) = 번들/캐시 최적화

**개발 중 보조 도구** ([원본: docs/설계/20260630-성능감사-구현설명.md])
- `WebVitalsReporter` — Next.js 내장 `useReportWebVitals` 훅으로 개발 환경 콘솔에 실시간 지표 출력
- `@next/bundle-analyzer` 도입, `npm run analyze`로 번들 시각화
- 운영 환경은 기존 Sentry 설정(`tracesSampleRate: 1`)으로 Web Vitals 자동 수집 활용

**작업 흐름**: ① 성능검사 문서에 페이지별 PSI 측정 기록 → ② README 통합 현황표로 취합 → ③ P0/P1 문제를 성능개선 문서로 옮겨 원인 분석·개선·전후 비교 기록.

담당 분배는 임태형(중고거래/내정보/찜/알림), 박소유(메인/글쓰기/칸토고/약관/구인구직), 김도혁(로그인/회원가입/부동산/프로필)으로 나눴으나, 실제로는 이동근을 포함해 서로 겹쳐 작업한 페이지도 있습니다.
