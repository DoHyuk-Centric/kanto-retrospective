## 성능검사 결과

`/terms/service`(임태형), `/terms/privacy`(김도혁), `/terms/youth`·`/terms/payment`·`/terms/policy`(박소유) 다섯 페이지 모두 비슷한 패턴을 보였습니다 — 모바일 Performance 72~81, **모바일 LCP 4.8~5.1s 🔴**, 데스크탑은 93~98점으로 우수. 렌더링 차단 요청(모바일 590~600ms), 미사용 JS 185~186KiB이 공통으로 지적됐습니다.

`termsservice.md`(임태형)에서 코드 레벨 원인을 정확히 짚었습니다.

> "콘텐츠 전체가 클라이언트에서만 렌더링됨. `TermsClientWrapper.tsx`가 `next/dynamic(..., { ssr: false })`로 `TermsClientContent.tsx`를 불러오고, 이 컴포넌트는 마운트 후 `useEffect`에서 `/api/terms`를 fetch해 내용을 채움 → HTML(서버) → JS 번들 다운로드 → hydrate → API fetch → 렌더링의 직렬 체인."

`localStorage` 캐시(10시간)가 있었지만 "재방문 시에는 도움이 되지만, 최초 방문(Lighthouse 측정 시나리오)에는 적용되지 않아 PSI 점수 개선에는 효과 없음"이라는 점도 함께 기록됐습니다.

## 개선 — SSR + ISR로 전환

[원본: docs/성능/성능개선/terms-SSR전환.md, 담당 박소유, 대상 5개 페이지 전체, P0, 완료]

세 가지 대안을 검토했습니다.
- 클라이언트 fetch + SWR/React Query — 최초 방문에는 여전히 직렬 체인이 남아 LCP가 개선되지 않아 기각
- SSR 단독(revalidate 없음) — Notion API 응답이 지연되면 매 방문마다 지연이 발생해 기각
- **SSR + ISR(`revalidate = 86400`) 채택** — 서버가 Notion 데이터로 채운 HTML을 내려주므로 직렬 체인이 사라지고, Notion API는 하루 한 번만 호출되고 이후 방문은 캐시된 HTML을 즉시 서빙합니다.

`TermsClientWrapper.tsx`(클라이언트 격리 장치 자체가 불필요해짐)와 `TermsClientContent.tsx`(useEffect 기반 fetch·로딩/에러 UI·localStorage 캐시 로직 전체)를 삭제하고, 각 역할을 `page.tsx`의 서버 fetch, `loading.tsx`, ISR로 옮겼습니다. 이는 원래 이 구조를 클라이언트 캐싱으로 설계했던 [Terms Notion 캐싱 설계](/features/김도혁)의 트레이드오프가 성능검사에서 실제로 드러나, 다시 서버 렌더링으로 되돌아온 사례이기도 합니다.
