## 성능검사 결과

**목록 `/usedgoods`** ([원본: docs/성능/성능검사/usedgoods.md], 측정 임태형): Performance 모바일 94 / 데스크탑 94, LCP 2.6s 🟡 / 0.7s 🟢. "모바일이 유일한 미달 지표 — 2.5s 목표를 근소하게 초과."

**상세 `/usedgoods/[id]`** ([원본: docs/성능/성능검사/usedgoodsid.md], 측정 임태형): Performance 81 / 88, LCP 3.1s 🟡 / 0.8s 🟢, TBT 430ms.

코드를 직접 확인해 원인을 진단한 부분이 특히 꼼꼼합니다.

> "`UsedGoodsDetail.tsx`가 통째로 `"use client"`이며, 좌표가 있는 매물은 `ApproxAreaMapWithProvider`가 진입 즉시 마운트되어 Google Maps JS SDK를 동기 로드·실행함."

> "코드 확인 결과: `ImageCarousel.tsx`는 이미 next/image + 첫 이미지 priority 적용이 되어 있어 LCP 이미지 자체 최적화는 양호. 병목은 이미지보다 지도 SDK·클라이언트 번들 실행(TBT) 쪽에 가깝다."

즉 다른 페이지들과 달리 이미지 최적화는 이미 잘 되어 있다는 걸 먼저 확인하고, 진짜 병목이 어디인지 코드 레벨에서 정확히 짚어낸 사례입니다. 중고거래 상세의 지도 지연 로딩 자체를 다루는 전용 개선 문서는 아직 없습니다.

목록 페이지의 이미지 전송 개선은 [공통 LCP 이미지 우선순위 개선](/performance/공통-lcp-이미지)에서 다뤘습니다.
