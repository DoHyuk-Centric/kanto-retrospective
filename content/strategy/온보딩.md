신규 팀원 온보딩 가이드는 실제로 동작하는 중고거래(usedgoods) 흐름 하나를 표준 모델로 삼아 "Kanto에서 화면 하나를 처음부터 끝까지 만드는 법"을 안내합니다.

## 구성에서 팀 문화가 드러나는 부분

1. **프로젝트 구조 한눈에 보기**: 라우트 그룹, `_components`, `services/`, `hooks/`, `constants/routes.ts` 등 폴더별 역할 설명
2. **원칙 2가지**: "읽기는 가능한 한 서버 컴포넌트에서, services 함수를 통해" / "클릭·입력 같은 상호작용은 `use client` 컴포넌트에서"
3. **표준 쓰기 패턴 명문화**: 낙관적 업데이트를 "기억 → 먼저 반영 → 실패 시 원복" 세 박자로 정형화하고, `LikeButton`을 표준 예시로 제시
4. **공통 컴포넌트 재사용 문화**: "화면을 만들다 필요한 게 있으면 새로 만들기 전에 `src/components/common`부터 봐주세요"
5. **경로 상수화 관행**: `ROUTES` 상수를 문자열 리터럴 대신 사용하도록 강제
6. **6단계 새 기능 추가 체크리스트**: services 함수 → 목록 페이지 → 목록/카드 컴포넌트 → 상세 페이지 → 상호작용 → 경로/공통 정리 → tsc 확인 후 PR

이 문서는 리뷰 지적사항(라우트 오타, RLS, 낙관적 업데이트 롤백 누락 등)이 반복되며 쌓인 교훈을 신규 팀원 온보딩용으로 체계화한 성격이 강합니다. [문제해결](/troubleshooting) 페이지에 나오는 실제 버그들 — 롤백 누락, 경로 하드코딩, 컴포넌트 중복 구현 — 이 왜 여기 체크리스트 항목으로 들어갔는지 짝지어 보면, 팀이 시행착오를 겪고 그걸 문서화해 다음 사람에게 전달하는 과정을 그대로 볼 수 있습니다.
