433건의 커밋 중 관리자 페이지·구인구직·중고거래를 오가며 가장 많이 마주친 건 "겉으로는 사소해 보이는데 파고들면 구조적인" 버그였습니다. 아래는 그중 기록이 남아있는 사례들입니다.

## 헤더가 스크롤할 때 이상하게 움직인다 — Opus 플랜 모드로 재설계

`2026-06-17` · [원본: docs/작업기록/20260617-header-hide-on-scroll.md]

모바일 헤더를 스크롤에 따라 숨기고 보여주는 기능이 있었는데, 실제 코드는 `IntersectionObserver`로 "최상단 근처인지 아닌지"만 이분법으로 판단하고 있었습니다. 그 결과 스크롤을 위로 올려도 최상단에 닿기 전까지는 헤더가 다시 나타나지 않는, 흔한 모바일 웹앱과는 다른 어색한 동작이 됐습니다.

`#header-sentinel` 요소가 뷰포트에 보이는지만 감지하는 구조적 한계였기 때문에, 패치보다는 재설계가 필요하다고 판단했습니다. **Claude Opus를 플랜 모드로 띄워 재설계를 요청**했고, `IntersectionObserver`를 걷어내고 `prevScrollY` ref 기반 스크롤 방향 감지 로직으로 교체했습니다(`ScrollToTop.tsx`에 이미 있던 패턴을 재사용). 결과적으로 "위로 스크롤하면 즉시 표시, 아래로 스크롤하면 숨김, 최상단 48px 이내는 항상 표시"라는 원래 문서가 설명하던 동작과 실제 코드가 처음으로 일치하게 되었습니다.

## 그 헤더 수정이 다음날 로그아웃 모달을 망가뜨렸다

`2026-06-17` · [원본: docs/작업기록/20260617-logout-modal-위치-버그-수정.md]

로그아웃 확인창이 화면 정중앙이 아니라 헤더 안에 갇혀 표시되는 버그가 발견됐습니다. 원인을 추적해보니 바로 전날 커밋(`fc9b0cb`, 위 헤더 스크롤 숨김 작업)에서 헤더에 `transition-transform` + `translate-y-*`를 추가한 게 원인이었습니다. `position: fixed` 요소는 조상 중 `transform`이 걸린 요소가 있으면 뷰포트가 아니라 그 조상 기준으로 위치가 잡힌다는 CSS 규칙 때문에, `<header>` 안에서 렌더링되던 `ConfirmModal`이 함께 영향을 받은 것입니다.

Fragment로 감싸 모달을 헤더 밖으로 빼는 방법(변경 최소)도 검토했지만, 그러면 "transform 조상 안에 fixed 모달을 두면 안 된다"는 제약을 다른 호출부(`DeleteButton` 등)가 매번 기억해야 합니다. 대신 `ConfirmModal`을 `createPortal`로 `document.body` 직속에 렌더링하도록 바꿔, 버그의 원인 유형 자체를 컴포넌트 레벨에서 없앴습니다. 어제 고친 게 오늘 새 버그를 낳는 경험을 하면서, "국소 수정"과 "원인 유형 제거"의 차이를 체감한 사례입니다.

## 채팅방 배지 숫자가 안 사라지고, 누르지도 않았는데 채팅방이 생긴다

`2026-06-18` · [원본: docs/작업기록/20260618-chat-unread-bug-도혁.md]

두 가지 버그가 겹쳐 있었습니다. 하나는 채팅방을 나가거나 새로고침해도 읽지 않은 메시지 배지가 사라지지 않는 것, 다른 하나는 "채팅하기" 버튼만 눌러도 메시지 없이 빈 채팅방이 목록에 생성되는 것이었습니다.

원인은 각각 달랐습니다. `markChatReadAction`이 마운트 시 딱 한 번만 호출되고 채팅방에 머무는 동안 새로 들어오는 메시지가 unread 카운터를 계속 올려도 리셋하는 로직이 없었습니다. 그리고 `postChat.ts`는 버튼 클릭 시점에 메시지 전송 여부와 무관하게 즉시 `chats` 테이블에 INSERT하고 있었습니다 — "채팅방 생성 시점이 '버튼 클릭'이 아니라 '첫 메시지 전송'이어야 한다"는 게 핵심 진단이었습니다.

읽음 처리는 언마운트 시에도 재호출하고 로컬 상태를 낙관적으로 즉시 0으로 갱신하도록 고쳤습니다. 채팅방 생성은 `postChat.ts`를 조회 전용으로 바꾸고, `chatStore`에 `PendingNewChat` 상태를 신설해 DB에 채팅방이 없어도 UI를 열 수 있게 한 뒤, 첫 메시지 전송 시점에 `createChatAndSendAction`(레이스 컨디션 방어 포함)이 채팅방 생성과 메시지 저장을 원자적으로 처리하도록 재설계했습니다.

## 어드민 페이지 3개가 890줄짜리 파일 하나에 다 들어있었다

`2026-06-26` · [원본: docs/작업기록/20260626-권한관리-컴포넌트분리.md, 20260626-모니터링탭-컴포넌트분리.md, 20260626-콘텐츠관리-리팩토링.md]

같은 날 세 개의 어드민 페이지를 연달아 리팩토링했습니다. `PermissionsClient.tsx`(894줄)는 어드민 목록·팀 목록·어드민 추가 세 탭을 한 파일에서 다뤘는데, 모든 액션이 `useTransition` 하나를 공유해서 "어드민 삭제 중에는 팀 생성 버튼도 비활성화"되는 문제가 있었습니다. 콘텐츠 관리 페이지(780줄)는 `useState` 16개, 핸들러 10개 이상이 한 파일에 몰려 있었고요.

세 곳 모두 같은 원칙으로 풀었습니다. 탭별로 컴포넌트를 쪼개고, TanStack Query(`useQuery`/`useMutation`)로 서버 상태를 옮겨 탭마다 독립적인 로딩 상태를 갖게 했습니다. 다만 모니터링 탭은 이미 `page.tsx`가 서버 컴포넌트에서 `Promise.all`로 데이터를 한 번에 받아 내려주는 구조라 TanStack Query가 불필요하다고 판단해 도입하지 않았습니다 — 패턴을 기계적으로 복사하지 않고 매번 "여기 필요한가"를 따로 물은 셈입니다. 콘텐츠 관리 페이지는 `page.tsx`가 780줄에서 65줄로 줄었습니다.

## Supabase 트래픽이 한도에 근접했다 — 이미지가 범인이었다

`2026-07-03` · [원본: docs/작업기록/20260703-supabase-이미지-트래픽-최적화.md]

Supabase 트래픽(egress) 사용량이 플랜 한도에 근접했다는 걸 발견하고 원인을 4가지로 특정했습니다. 원본 이미지(수 MB)를 그대로 업로드하고 있었고, 공용 `ImageWithFallback`에 `loading="eager"`가 하드코딩돼 lazy loading이 무력화돼 있었고, Storage 업로드 시 `cacheControl`을 지정하지 않아 기본 캐시가 1시간뿐이었고, 일부는 `<img>`를 직접 써서 next/image 캐시 자체를 우회하고 있었습니다.

canvas 기반으로 비율을 유지하며 축소하고 WebP(품질 0.8)로 변환하는 `optimizeImage.ts` 유틸을 새로 만들어, 게시물 이미지는 1600px, 아바타·로고는 512px로 상한을 걸었습니다. 업로드 캐시는 1년(`cacheControl: "31536000"`)으로 늘렸습니다. 저장 용량·업로드 트래픽 70~90% 절감을 기대하고 있고, 실제 효과는 배포 후 Supabase 대시보드의 Egress 그래프로 확인할 계획입니다 — 아직 "고쳤다"가 아니라 "고치고 지켜보는 중"인 사례입니다.

## 그 밖의 작업들

- **구인구직 카드 정보 보강 (1~4차, 2026-06-17~18)**: 처음엔 카드에 제목·회사명·급여·위치만 있어 정보가 부족했는데, 필드를 채우다 보니 이번엔 데스크탑에서 컬럼 폭을 고정값(`w-40`/`w-48`/`w-24`)으로 줘서 창이 좁아지면 본문 텍스트만 무너지는 문제가 생겼습니다. `flex-2:flex-4:flex-3` 비율 + `min-w-0`으로 컬럼이 함께 유동적으로 줄어들게 고쳤고, 이후 사용자 피드백에 따라 우대사항 컬럼을 근무시간으로 교체하는 DB 마이그레이션까지 진행했습니다. "지원자 현황"은 채팅=지원이라는 모델상 별도 설계가 필요해 의도적으로 범위에서 뺐습니다.
- **검색바/필터 드롭다운 개선 (2026-06-17)**: 데스크탑 지역 드롭다운이 `fixed` + `getBoundingClientRect`로 위치를 계산해 스크롤해도 버튼과 함께 움직이지 않는 문제를 `absolute top-full` 방식으로 교체.
- **CSS 클래스 통합 검토, 접근성 개선 가이드 적용, 예약중/판매완료 상태 기능, 메인페이지 UI 조정, URL id 암호화** 등도 함께 진행했습니다.
