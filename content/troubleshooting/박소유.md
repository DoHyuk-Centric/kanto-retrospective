팀장으로서 PR을 가장 많이 병합했고, 그만큼 "누군가의 변경이 다른 곳에서 어떻게 어긋나는지"를 자주 발견하는 위치에 있었습니다. 아래는 그 중 문서로 남은 사례입니다.

## 하루 동안 발견한 것: 히스토리 오염, 민감정보 노출, 찜 카운트가 두 배로

`2026-06-17` · [원본: docs/작업기록/20260617-bugSoyu.md]

한 세션에서 성격이 전혀 다른 버그 네 개를 연달아 짚었습니다.

첫째, 게시글 수정 후 "목록으로"를 누르면 수정 페이지로 되돌아가는 문제 — `router.push()`가 히스토리를 계속 쌓고 있었습니다. `router.replace()`로 바꾸고, 상세 페이지의 "목록으로"는 `router.back()` 대신 명시적으로 `router.push("/usedgoods")`로 바꿨습니다.

둘째, 더 심각한 문제였습니다. `users(*)` JOIN 쿼리가 email·provider·updated_at 같은 민감 정보를 클라이언트에 그대로 노출하고 있었습니다. 모든 상세 페이지 쿼리에서 필요한 컬럼(id, name, avatar_url, auth_id, role, post_count, created_at)만 명시적으로 select하도록 고쳤습니다.

셋째, `posts.users`가 null일 수 있는데 옵셔널 체이닝 없이 접근하는 코드가 있어 런타임 에러 위험이 있었습니다. 타입을 `RentSellerInfo | null`로 바꾸고 전체에 optional chaining을 적용했습니다.

넷째, 가장 까다로웠던 건 찜 카운트였습니다. `common_likes` INSERT가 403으로 실패하고, 찜 카운트가 실제 찜 개수와 어긋나 있었습니다. 파고들어 보니 DB 트리거 두 개(`trg_like_count`, `trg_update_like_count`)가 중복으로 같은 함수를 호출해서 `users.like_count`가 두 배로 올라가고 있었고, 정작 `posts.like_count`(글이 받은 찜 수)를 갱신하는 트리거는 아예 없었습니다. 403 에러는 `notify_on_like()` 트리거 함수가 `SECURITY DEFINER` 없이 실행돼 notifications 테이블에 쓸 권한이 없었던 게 원인이었습니다. 중복 트리거를 제거하고 `update_post_like_count()`를 새로 만들어 기존 데이터를 재계산했습니다.

한 가지는 미해결로 남겼습니다 — `ReportModal`의 이중 INSERT 버그를 고치려고 제안했지만 세션 중 거절당해 다음에 재확인이 필요하다고 문서에 그대로 적어뒀습니다. 다음 사람을 위해 폴더 구조와 컨벤션, 결정사항까지 문서 하나에 총정리해둔 것도 이 세션의 특징입니다.

## 채팅방 진입할 때마다 DB를 세 번 왕복하고 있었다

`2026-06-23` · [원본: docs/작업기록/20260623-채팅성능개선-박소유.md]

채팅 관련 성능을 뜯어보면서 네 가지 낭비를 찾아냈습니다. 채팅방 진입 시 chat 조회, posts 조회, used_goods 조회가 따로따로 실행돼 DB 왕복이 3회 발생했습니다. 읽음 처리는 `useChatMessages`/`ChatRoomClient`의 마운트·언마운트 세 곳에서 분산 실행돼 messages와 chats.unread가 어긋날 수 있는 구조였습니다. 메시지를 조회할 때마다 대부분 null인 `transactions(*)` 조인이 매번 평가됐고, 채팅 목록 실시간 구독은 필터 없이 전체 `chats` 테이블 변경을 구독해서 내 채팅이 아닌 이벤트까지 전달받고 있었습니다.

각 문제를 쿼리 단위로 분해해서 원인을 특정한 뒤, 인덱스 4개를 새로 추가하고 채팅방 조회를 단일 임베드 쿼리로 통합해 왕복을 3회에서 1회로 줄였습니다. Supabase RPC `mark_chat_read`를 만들어 messages와 chats.unread를 트랜잭션 안에서 한 번에 처리하도록 하고, 분산돼 있던 읽음 처리 로직을 걷어냈습니다. 실시간 구독도 user_id 기준 필터 채널 2개로 분리해 불필요한 이벤트 수신을 없앴습니다.

## 메시지 전송 버튼 하나에 서버 왕복이 세 번

`2026-06-23` · [원본: docs/작업기록/20260623-리뷰개선-박소유.md]

메시지 전송 버튼을 누를 때마다 클라이언트가 정지 확인, 차단 확인, 메시지 전송까지 서버 액션을 3번 연속 호출하고 있었습니다. 일반적인 전송 상황에서도 매번 불필요한 왕복이 발생하는 구조였습니다. 두 검증 호출을 제거하고 `sendMessageAction` 내부에서 같은 검증을 수행하도록 옮겨, 왕복을 3회에서 1회로 줄이면서 동시에 클라이언트에서 우회할 수 없는 서버 검증으로 강화했습니다. 채팅방 로딩 중에는 "loading" 텍스트만 있던 걸 실제 레이아웃(헤더/메시지/입력창) 구조와 동일한 Skeleton 컴포넌트로 바꿨습니다.

## 컴포넌트 재사용의 대가: 유령 페이지와 하드코딩된 경로

`2026-06-14` · [원본: docs/작업기록/20260614-컴포넌트-개선-박소유.md]

헤더의 연필 아이콘(글쓰기 버튼)을 각 카테고리 목록 페이지의 글쓰기 버튼으로 통합하기로 했는데, 헤더에서 제거하는 부분만 먼저 논의되면서 실제로는 아직 create 페이지가 연결되지 않은 "유령 페이지"가 잠깐 생겼습니다. `EditButton`/`DeleteButton`도 `/usedgoods/{id}/edit` 같은 경로를 하드코딩하고 있어서 다른 도메인에서 재사용할 수 없었고, 작성자 본인 확인(`isOwner`) 로직도 각 상세 컴포넌트마다 중복 구현돼 있었습니다.

`editPath`/`redirectPath`를 prop으로 외부 주입하도록 바꾸고, `VerifyAuthor` 공통 컴포넌트로 작성자 확인 로직을 통합했습니다. job/rental 상세에도 수정·삭제 버튼을 추가했고요. rental 수정 페이지는 "글쓰기 폼 자체가 없어서" 의도적으로 미완료 상태로 남겨두고, 글쓰기 구현 이후 같은 패턴으로 추가하기로 문서에 명시했습니다.

## 결제: 안 되던 실제 송금, 그리고 중복 클릭

[원본: docs/작업기록/20260625-결제흐름변경.md, 20260625-결제보안강화.md, 20260625-관리자결제관리및만료처리.md]

에스크로 결제에서 `released`로 상태가 바뀌어도 판매자에게 실제 송금이 발생하지 않는 문제를 발견했습니다. Xendit Disbursement API 연동 자체가 없었던 것입니다. 은행 계좌 컬럼을 추가하고 프로필에 정산 계좌 등록 UI를 만든 뒤, `confirmReceiptAction`이 released 전환 전 판매자 계좌를 조회하고 없으면 차단, 있으면 `createDisbursement()`를 호출하도록 연결했습니다.

곧이어 더 급한 문제가 나왔습니다. 구매자가 "수령 확인" 버튼을 빠르게 두 번 누르면 disbursement가 중복 호출돼 판매자에게 돈이 두 번 지급될 수 있는 레이스 컨디션이었습니다. PostgreSQL의 atomic `UPDATE ... WHERE status='paid'`를 활용한 `claimTransactionRelease()`로, 첫 요청만 갱신에 성공하고 이후 요청은 즉시 실패하도록 만들었습니다.

로컬 개발 환경에서는 Xendit webhook이 동작하지 않아 24시간 지난 pending 결제가 만료 처리되지 않는 문제도 있었는데, `created_at + 24h < now` 체크로 UI 레벨에서 만료 여부를 판단하도록 임시 대응하고, 어드민 결제 관리 페이지를 새로 만들었습니다.
