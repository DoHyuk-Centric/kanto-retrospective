작업 로그 파일이 이름으로 남은 다른 팀원들에 비해 적지만, 사이트 전체를 훑는 성능 감사와 알림 시스템의 숨은 버그처럼 범위가 넓은 작업을 맡았습니다.

## "새로고침하면 다시 안읽음이 된다" — NULL이 숨겨온 버그

`2026-06-23` · [원본: docs/작업기록/20260623-notification-bug.md]

알림 목록에서 "모두 읽음"을 누르고 새로고침하면 다시 미읽음 상태로 돌아가는 버그가 있었습니다. 이상하게도 알림을 하나씩 클릭해서 읽으면 정상이었습니다. 헤더 알림벨 카운트도 전체보기에서 모두읽기를 처리해도 줄어들지 않았고요.

원인은 SQL의 3치 논리(three-valued logic)에 있었습니다. `.eq("is_read", false)` 필터는 `is_read IS NULL`인 행을 매칭하지 못합니다. NULL인 행은 업데이트되지 않고 새로고침 시 부활(`!null`이 JS에서는 `true`로 평가되니까)하는 것이었습니다. 실측해보니 전체 353행 중 무려 93%(329행)가 `created_at IS NULL`이었고, 관리자 정지/해제 알림을 만드는 INSERT가 애초에 이 컬럼들을 채우지 않고 있었다는 게 근본 원인이었습니다. 여기서 그치지 않고 `.order("created_at", desc)`가 Postgres에서는 NULLS FIRST라서, NULL 행이 상단으로 몰려 `limit(50)`에서 정작 유효한 최신 행이 잘려나갈 수 있는 잠복 위험까지 확인했습니다.

`.eq("is_read", false)` 조건을 제거해 NULL 행도 포함해서 갱신되도록 고치고, 마이그레이션 두 건으로 NULL을 backfill한 뒤 컬럼에 DEFAULT/NOT NULL을 부여했습니다. 흥미로운 건 문서에 스스로 남긴 정정 기록입니다 — "어제(06-22) `--continue`로 대화를 이어가는 과정에서 요약되면서 지적했던 항목 번호 목록 원문이 증발해, 다시 증발하지 않도록 지금 기록을 남긴다"고 적어뒀습니다. AI와 나눈 대화가 요약(compact)되면서 세부 내용이 사라지는 걸 실제로 겪고, 그걸 막기 위해 의식적으로 문서화한 셈입니다.

## 회원가입할 때 영어 에러가 그대로 노출됐다

`2026-06-19` · [원본: docs/작업기록/20260619-작업기록.md]

한 세션에서 세 가지를 고쳤습니다. 중복 이메일로 가입하면 한국어 안내 대신 "User already registered"라는 영어 원문이 그대로 노출되고 있었습니다 — `enable_confirmations = false` 설정 때문에 `error` 분기로 떨어지면서 원문 메시지가 그대로 나온 것이었습니다. `error.code === "user_already_exists"` 분기를 추가해 해결했습니다.

본인인증 API는 입력한 이메일을 가입 이메일과 대조하지 않아서, 다른 이메일로도 인증번호를 받을 수 있는 문제가 있었습니다. 인증 코드 생성 전에 이메일 일치 검사를 추가했습니다.

세 번째는 국제화 설정 실수였습니다. 필리핀어 로케일 코드로 `ta`를 쓰고 있었는데, BCP-47 기준으로 `ta`는 타밀어이고 필리핀어는 `fil`이어야 했습니다. `messages/ta.json`을 `fil.json`으로 옮기고 관련 설정을 전면 교체했습니다. 전수 점검하다가 프로필의 "언어 설정" 선택기가 로컬 state만 바꿀 뿐 `setLocaleCookie`/`refresh`를 호출하지 않아 애초에 작동하지 않는 UI였다는 것도 발견해서, 헤더 스위처로 일원화하고 아예 삭제했습니다.

## 사이트 전체 성능 감사 — "작아서 느린 게 아니라 왕복이 많아서 느리다"

`2026-06-23` · [원본: docs/작업기록/20260623-performance-audit.md]

채팅을 제외한 user/admin 전반의 성능을 감사했습니다. 상세 페이지는 독립적인 쿼리를 순차 `await`로 실행해서 상세 페이지 하나당 최대 5~7회의 직렬 DB 왕복(싱가포르 리전)이 발생하고 있었고, 목록 페이지는 `.limit()` 없이 활성 게시물 전체를 로드한 뒤 JS `slice()`로 페이지네이션과 필터링을 처리하고 있었습니다. FK 인덱스도 다수 누락돼 있었고, RLS의 `auth.uid()`가 매 행마다 재평가되는 비효율도 있었습니다.

여기서 눈에 띄는 건 진단 범위를 스스로 한정한 점입니다 — "현재 데이터 규모(posts 71건 등)가 작아서, 체감 느림의 주범은 캐싱 부재라기보다 순차 왕복 횟수"라고 명시했습니다. 코드 정독과 Supabase performance advisor, 원격 카운트 쿼리로 근거를 확보한 뒤 우선순위를 정했습니다: 왕복 병렬화 + auth id 캐시 → 인덱스 + RLS → 진짜 페이지네이션/필터 → 캐싱 순.

`Promise.all` 병렬화와 `after()`로 조회수 증가를 응답 이후 비차단 처리했고, `getAuthUser`/`getCurrentUserId`에 React `cache()`를 적용해 중복 조회를 없앴습니다. 목록 3종은 `!inner` 조인 + `.range()` + `count:"exact"`로 진짜 DB 페이지네이션으로 전환했습니다.

캐싱은 의도적으로 보류했습니다. "조사 결과 안전하게 출시 가능한 코드 캐싱이 없음을 확인해서, cosmetic한 변경으로 흉내내지 않고 설계만 남긴다"는 이유였습니다 — 게시글 변경 지점이 여러 곳에 흩어져 있어 태그 무효화를 하나라도 놓치면 가격·판매완료·예약 상태가 오래된 값으로 보이는, 거래 플랫폼에서는 치명적인 버그가 될 수 있다는 판단이었습니다. 프로덕션 DB에 직접 영향을 주는 마이그레이션은 "AI 세션의 자동 실행 분류기가 모호한 지시라고 판단해 1차로 거부했고, 이후 명시적으로 승인 질문을 받은 뒤에야 적용했다"고 기록돼 있습니다 — 위험한 작업 앞에서 안전장치가 실제로 작동한 과정을 그대로 남긴 셈입니다.
