채팅의 실시간성 문제와 칸토Go(위치 기반 번개모임)라는, 팀에서 가장 예측 불가능한 두 영역을 주로 맡았습니다. 둘 다 "재현하면 사라지고, 안 재현하면 나타나는" 부류의 버그가 많았습니다.

## 알림이 쌓이고, 보고 있는데도 알림이 온다

`2026-06-18` · [원본: docs/작업기록/20260618-chat-notification-dedupe.md]

채팅 메시지가 쌓일 때마다 알림이 계속 새로 INSERT되어서, 안 읽은 메시지가 여러 개면 알림 목록에 같은 채팅방 알림이 여러 줄로 누적되는 문제가 있었습니다. 더 이상한 건 상대가 지금 그 채팅방을 보고 있는 중에도 알림이 발송된다는 점이었습니다. 알림 트리거가 "상대가 지금 이 채팅방을 보고 있는지" 판단할 방법 자체가 없었던 것입니다(presence 부재).

`chats` 테이블에 `user_id_1_active`/`user_id_2_active` presence 컬럼을 추가하고, 본인 슬롯만 갱신하도록 security definer로 제한한 `set_chat_active` RPC를 만들었습니다. 알림 트리거는 시스템 메시지는 건너뛰고, 수신자가 active 상태면 건너뛰고, 같은 채팅방에 이미 미확인 알림이 있으면 새로 INSERT하는 대신 UPDATE로 합치도록 바꿨습니다.

검증하다가 새 버그가 튀어나왔습니다. 알림 realtime 채널명을 `notifications:${userId}:${Date.now()}`처럼 매번 달라지게 만들어뒀는데, 헤더의 `NotificationBell`과 `/notifications` 페이지가 동시에 `useNotifications()`를 호출하면 채널명이 겹치면서 "cannot add postgres_changes callbacks... after subscribe()" 런타임 에러가 새로 발생했습니다. `useId()` 기반으로 인스턴스별 고유 suffix를 붙여 해결했습니다. 하나를 고치다가 다른 하나가 깨지는, 실시간 기능 특유의 연쇄 반응을 겪은 사례입니다.

## "1"이 깜빡이거나, 늦게 뜨거나 — 두 요구가 충돌한다

`2026-06-18` · [원본: docs/작업기록/20260618-chat-read-timing-임태린.md]

상대가 채팅방을 보고 있어서 바로 읽는 상황에서도 안읽음 표시 "1"이 잠깐 나타났다 사라지는 깜빡임이 있었습니다. 1차로 수정했더니 이번엔 반대 상황에서 회귀가 생겼습니다 — 상대가 정말 못 읽었을 때는 "1"이 즉시 나타나지 않고 한 박자 늦게 뜨게 된 것입니다.

문제를 정확히 진단하면 이렇습니다: 낙관적 메시지가 `is_read: false`로 즉시 추가되어 "1"이 먼저 보이고, 상대가 보고 있으면 곧 UPDATE가 왕복(수백 ms)해 사라지는 과정에서 깜빡임이 생깁니다. 그런데 상대가 지금 보고 있는지 알 방법이 없으니 "무조건 N초 기다렸다 표시"하는 방식으로는 "깜빡임 제거"와 "못 읽었으면 즉시 표시"라는 두 요구를 동시에 만족시킬 수 없습니다.

Supabase Realtime Presence를 도입해 상대의 실시간 접속 여부(`partnerOnline`)를 계산하고, "1" 렌더링을 `UnreadMark`라는 별도 컴포넌트로 분리했습니다. 오프라인이면 즉시 표시하고, 온라인이면 실제 읽음 처리가 도착하는 시간(약 1초) 만큼만 지연 표시해서, 그 사이 진짜 읽음이 오면 아예 렌더링되지 않고 사라지게 만들었습니다. React 19의 `react-hooks/purity`, `react-hooks/set-state-in-effect` 규칙을 지키기 위해 시간 계산은 `useEffect`에서, `setState`는 항상 `setTimeout` 콜백 안에서만 호출하도록 신경 썼습니다.

## 차단했는데 메시지가 간다 — RLS의 함정

`2026-06-18` · [원본: docs/작업기록/20260618-유저신고차단.md]

채팅 유저 신고·차단 기능을 처음 만들면서 두 가지 버그를 만났습니다. 하나는 차단 검증이 항상 실패하는 것, 다른 하나는 메시지가 잠깐 보였다가 사라지는 UX 버그였습니다.

첫 번째 원인이 특히 인상적이었습니다. `isBlockedPair`가 일반 서버 클라이언트(RLS 적용)로 `user_blocks`를 조회했는데, RLS 정책상 `blocker_id = my_user_id()`인 행만 보입니다. 그런데 **차단당한 쪽이 메시지를 보낼 때는 그 차단 row 자체가 그 사람 눈에는 보이지 않으니**, 검증 자체가 무력화되는 것이었습니다. "차단당한 쪽은 자기 세션으로 이 테이블을 조회할 수 없다"는 RLS 설계의 함정을, 향후 유사한 검증 로직에서 반드시 admin 클라이언트를 써야 한다는 교훈으로 문서에 명시적으로 남겼습니다. `isBlockedPair`를 `createAdminClient()`로 바꿔 해결했습니다.

두 번째는 메시지를 먼저 낙관적으로 화면에 그린 뒤 서버 액션 내부에서 차단 여부를 검사해서, 차단된 경우 메시지가 잠깐 보였다가 삭제되는 것이었습니다. 낙관적 메시지를 추가하기 전에 먼저 차단 여부를 확인하도록 순서를 바꿨습니다.

## 채팅 위젯 위에서 스크롤하면 뒷배경이 같이 움직인다

`2026-06-19` · [원본: docs/작업기록/20260619-chat-scroll-overflow-임태린.md]

플로팅 채팅 위젯이 열려 있을 때 위젯 위에서 휠을 굴리면 위젯이 아니라 뒷배경 페이지가 같이 스크롤됐습니다. 스크롤 경계(맨 위/아래)에 닿은 뒤에도, 심지어 스크롤이 없는 헤더나 입력창 위에서도, 내용이 짧아 스크롤 자체가 없는 목록 위에서도 배경이 움직였습니다.

데스크탑에는 배경 스크롤을 막는 처리가 아예 없었습니다. 모바일에서 쓰던 `document.body.style.overflow = "hidden"`은 body 자체의 스크롤만 막을 뿐, 위젯 내부 스크롤이 경계에 도달했을 때 부모로 전파되는 "스크롤 체이닝"은 막지 못한다는 걸 정확히 짚었습니다. 그리고 React의 `onWheel` prop은 브라우저 레벨에서 passive 리스너로 등록되기 때문에 `preventDefault()`가 아예 동작하지 않는다는 사실도 확인했습니다.

CSS `overscroll-behavior: contain`을 리스트 요소들에 적용하고, 위젯 루트에 `ref`를 걸어 `addEventListener("wheel", handler, { passive: false })`로 직접 등록했습니다. 스크롤 여지가 있으면 허용하고, 경계나 비스크롤 영역이면 `preventDefault()`로 전파를 막았습니다. 기존 모바일 body-lock 로직은 그대로 두고 새 wheel 핸들러를 더해서, 데스크탑과 모바일 모두를 안전하게 보완했습니다.

## 알림은 실시간인데 배너는 실시간이 아니다

`2026-06-23` · [원본: docs/작업기록/20260623-제재배너-실시간반영.md]

관리자가 유저를 제재하거나 해제하면 알림은 즉시 뜨는데, 상단 제재 배너는 새로고침을 해야 나타나거나 사라졌습니다. 심지어 뒤로가기/앞으로가기로는 새로고침조차 안 되니 배너가 아예 갱신되지 않았습니다.

"알림은 실시간인데 배너는 아니다"와 "새로고침은 되는데 뒤로/앞으로가기는 안 된다"는 두 단서를 따라갔습니다. 배너는 `useAuthStore`의 `user.suspended_until`만 보는데, `authStore`는 `onAuthStateChange` 이벤트(로그인/로그아웃/토큰갱신/최초 세션 로드) 때만 users 테이블을 다시 조회하며, 이건 DB의 users 행 변경과는 아무 상관이 없다는 걸 확인했습니다. `users` 테이블에는 realtime 구독이 전혀 없었던 게 근본 원인이었습니다. 새로고침은 앱이 재마운트되어 `INITIAL_SESSION`이 발생하지만, 클라이언트 라우팅(뒤로/앞으로가기)은 재마운트가 없어 그 이벤트조차 발생하지 않는다는 것까지 규명했습니다.

두 가지 해법을 놓고 비교했습니다: A안은 users 테이블 전체에 realtime 구독을 거는 것, B안은 알림 realtime에 편승해서 refetch만 하는 것. **B안을 선택**했는데, 제재/해제 액션이 항상 suspension 알림을 함께 insert하니 원하는 트리거와 정확히 1:1로 대응하고, 이미 검증된 알림 인프라를 재사용할 수 있고, users 전체를 broadcast했을 때 생기는 민감정보 노출 확대 리스크가 없기 때문이었습니다. `useAuthInit.ts`에 두 번째 useEffect를 추가해 `common_notifications` INSERT를 구독하다가 `type === 'suspension'` 알림이 도착하면 users 행을 refetch하도록 만들었습니다.

## 그 밖: 이미지 검열 AI, 칸토Go, 거래지역 세분화

이미지 검열 AI 기능은 설계가 사용자와의 대화를 거치며 네 번이나 바뀌었습니다. 독립 마이크로서비스(Hono+Docker) → API 라우트 내장으로 축소, fail-open → fail-closed로 전환(미검수 이미지가 공개되는 걸 사용자가 거부), OpenAI 폴백 검토 후 결제 한도 문제로 폐기 → Groq로 교체, Gemini+Groq 이중화 후 무료 티어 한도 문제로 Groq 단일 제공자로 최종 단순화까지 — 자세한 내용은 [기능구현](/features/임태형) 페이지에 있습니다.

칸토Go(위치 기반 번개모임)에서는 디자이너가 AI로 생성한 Next.js 코드 패키지를 그대로 포팅할 수 없었던 문제(UUID 가정 vs 실제 bigint, 존재하지 않는 import 경로 등), 실제 GCP 키 발급 후 브라우저 검증 단계에서 발견한 3건의 런타임 버그(리퍼러 제한, 마커 위치 어긋남, PostgREST가 간접 관계를 추론하지 못해 생긴 조회 실패)를 `.next/dev/logs/next-development.log`를 직접 읽어 정확한 에러 코드와 타임스탬프로 특정했습니다. 거래지역을 광역 enum에서 바랑가이(동 단위)로 세분화할 때는 "위치를 더 정확히 보여주고 싶다"와 "건물 단위로 노출되면 안 된다"는 상반된 요구 사이에서, 좌표를 화면에서만 흐리게 보여주는 게 아니라 저장 단계에서부터 반올림(`roundCoord`)하는 방식으로 풀었습니다.
